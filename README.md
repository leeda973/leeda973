#### 🔧 핵심 설계 — RBAC 기반 회사별 권한 커스터마이징 구조 설계

**문제**
- 인사·근태·급여·결재·평가·ESG 등 모든 HR 모듈 통합으로 관리 권한의 종류가 급증
- SaaS 특성상 회사마다 조직·직책별 권한 정책이 달라 하드코딩이 불가능하고 회사별 커스터마이징이 필수

**해결**
- 사용자가 아닌 역할(Role)에 권한을 부여하는 RBAC 도입
- 리소스 × 액션 × 범위 조합으로 권한을 세분화하고, 권한을 데이터로 관리해 회사별 커스터마이즈가 가능하도록 구조 설계
- `@CheckPermission` 어노테이션 + AOP로 비즈니스 로직과 권한 검사를 분리

**결과**
- 인사팀이 UI에서 직접 역할 생성·조합 → 회사별 권한 정책을 코드 수정 없이 자유롭게 수용
- `@CheckPermission` 한 줄로 권한 검사 적용
- 4개 MSA 전체에 일관된 권한 정책 유지 및 코드 중복 제거

> **핵심 수치** : 리소스 13종 × 액션 4종 × 범위 3종 → 최대 약 150여 개 권한 단위를 역할별로 자유롭게 조합

#### 🔧 Technical Decision — 동기/비동기 통신 선택 기준

**상황**
- MSA 아키텍처에서 전자결재 서비스가 member·salary 서버와 통신하는 상황이 다수 발생
- 결재 트랜잭션 내부에서 외부 서버 데이터가 필요한 경우와, 결재 승인 이후 후속 처리가 필요한 경우가 혼재

**고민**
- 동기 통신은 즉시 응답을 받을 수 있지만 외부 서비스 장애가 결재 트랜잭션 전체에 영향을 줄 수 있음
- 비동기 통신은 결합도를 낮추고 응답 속도를 확보할 수 있지만 즉시 응답이 필요한 상황에는 부적합
- 결재 트랜잭션 내에서 **즉시 필요한 데이터인지 여부**를 기준으로, 동기 호출은 최소화하고 나머지는 비동기로 분리하는 방향으로 설계

**선택과 이유**

✅ **동기 통신 (Feign Client)** — 결재 트랜잭션 내 즉시 결과가 필요한 경우에만 최소 적용
- 결재라인 편집 화면에서 즉시 후보자 리스트를 확인하는 경우
- 기안 생성 트랜잭션 안에서 기안자 직책·조직 정보를 함께 검증·저장하는 경우
- 승인 트랜잭션 안에서 결재자 서명 이미지를 즉시 문서에 저장하는 경우

✅ **비동기 통신 (Kafka)** — 결재 트랜잭션과 분리 가능한 후속 처리는 전부 비동기로 분리
- salary 서버 장애가 결재 승인 자체에 영향을 주면 안 되는 경우 (휴가 차감·근태 반영)
- 응답 시간에 영향을 주지 않아도 되는 후속 처리인 경우 (캘린더 등록·수당 변경)

#### 🛠️ 트러블슈팅 1 — S3 Presigned URL 도입으로 이미지 업로드 부하 제거

| 구분 | 내용 |
|---|---|
| **문제** | 피드·채팅 등 다수의 이미지 공유 기능에서 용량이 큰 파일 업로드·조회 시 모든 트래픽이 서버를 경유해 응답 지연과 부하 발생 |
| **원인** | 클라이언트 → 서버 → S3 구조로 이미지 데이터가 서버를 거쳐 전달되며, 서버가 불필요한 이미지 처리 병목 구간이 됨 |
| **해결** | S3 Presigned URL을 발급하여 클라이언트가 서버를 경유하지 않고 S3에 직접 업로드하도록 구조 변경 |
| **결과** | 서버의 이미지 처리 부하 제거 및 업로드 응답 속도 개선 |

```text
[변경 전] Client → Server(EC2) → S3 Bucket   (서버 경유, 병목 발생)
[변경 후] Client → (Presigned URL 발급) Server → Client → S3 Bucket  (서버 경유 없이 직접 업로드)
```

#### 🛠️ 트러블슈팅 2 — Redis 캐싱 도입으로 반복 DB 쿼리 제거

| 구분 | 내용 |
|---|---|
| **문제** | 크루 목록 조회는 메인 화면 진입마다 호출되는 빈도 높은 API로, 매 요청마다 DB SELECT 쿼리가 다건 발생하고 동일한 데이터임에도 불필요한 DB I/O가 반복적으로 누적 |
| **원인** | 조회 빈도는 높으나 실시간성 요구가 낮은 데이터임에도 캐싱 없이 모든 요청이 DB를 직접 조회하는 구조로 동작 |
| **해결** | `@Cacheable` 적용으로 첫 요청만 DB 조회하고 이후 Redis 캐시에서 즉시 응답. 크루 생성·수정·삭제 시 `@CacheEvict`로 즉시 무효화해 데이터 정합성 유지 |
| **결과** | 반복 조회 시 DB SELECT 쿼리 미발생, 불필요한 DB I/O 제거로 부하 감소 및 응답 처리 흐름 단순화 |

<br/>
